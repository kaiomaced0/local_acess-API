## Contexto
Hoje o schema é gerado por Hibernate + `import.sql`. Inviável para produção e para evoluir multitenancy.

## Objetivo
Adotar Flyway com migrações versionadas.

## Critérios de aceitação
- [x] `quarkus-flyway` adicionado ao `pom.xml`.
- [x] `quarkus.hibernate-orm.database.generation=validate` (ou `none`) em produção.
- [x] Migração inicial `V1__schema_inicial.sql` com todas as tabelas atuais.
- [x] `application.properties` configurado para rodar migrações no startup em dev/test.
- [x] README atualizado com instruções.

## Notas técnicas
- Preservar compatibilidade com MariaDB **e** Postgres (evitar dialetos).

## Resultado
- Arquivos alterados:
  - `pom.xml` — adicionada dependência `io.quarkus:quarkus-flyway`.
  - `src/main/resources/application.properties` — `database.generation=validate` em %dev/%test/%prod; `sql-load-script` desabilitado em %prod; Flyway configurado com `migrate-at-start`, `baseline-on-migrate` e `locations=classpath:db/migration`.
  - `src/main/resources/db/migration/V1__schema_inicial.sql` — novo; cria tabelas `Evento`, `EspacoEvento`, `TipoIngresso`, `TipoIngresso_EspacoEvento`, `Ingresso`, `Usuario`, `usuario_perfil` com PKs e FKs, usando `GENERATED BY DEFAULT AS IDENTITY` (padrão SQL suportado por MariaDB 10.3+ e PostgreSQL 10+).
  - `README.md` — nova seção "Migrações" explicando o fluxo.
- Build: `./mvnw.cmd compile -q` executado com sucesso (exit 0).
- Observações:
  - Os nomes de tabela/coluna seguem exatamente o que o Hibernate espera com a `PhysicalNamingStrategyStandardImpl` padrão (sem snake_case forçado). Identificadores são usados sem aspas no DDL para que tanto MariaDB (case-insensitive) quanto PostgreSQL (lowercase unquoted) validem o mapeamento gerado pelo Hibernate sem alterações no código Java.
  - `import.sql` continua sendo carregado apenas em `%dev` para seeds; em `%prod` foi explicitamente apontado para um arquivo inexistente (`no-file`) para não carregar.
  - A atividade 001 (multitenancy) vai precisar criar migrações `V2__...`/`V3__...` adicionando a tabela `Empresa` e FKs `empresa_id NOT NULL` em todas as tabelas de negócio. O schema atual NÃO inclui `Empresa` pois ainda não existe como entidade — isso é esperado e faz parte do trabalho da atividade 001.
  - Não foi testado em runtime (subir o app contra um banco real) — apenas compilação. Recomenda-se fazer um `quarkus:dev` contra MariaDB e PostgreSQL antes do deploy para confirmar que o Hibernate aceita o schema criado pelo Flyway.
